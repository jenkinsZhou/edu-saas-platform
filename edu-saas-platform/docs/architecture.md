# 架构方案

## 目标

建设一套支持多租户、高并发、可扩展的教培 SaaS 管理系统。当前优先建设 Web 管理端和后端能力，后续移动端通过统一 API 接入。

## 为什么先做模块化单体

教培系统早期的复杂度主要在业务建模、权限、租户隔离和数据关系。如果一开始就微服务化，会提前引入服务治理、分布式事务、链路追踪和部署复杂度。

因此第一阶段采用模块化单体：

- 模块边界清晰，方便后续拆服务。
- 同库事务简单，稳定性更高。
- 开发、调试、部署成本低。
- 业务快速迭代时更安全。

## 后端模块

```text
edu-common      公共响应、异常、审计字段、分页、枚举
edu-security    登录认证、JWT、权限校验、数据权限上下文
edu-tenant      租户、套餐、租户配置、租户主题
edu-system      账号、用户、角色、菜单、组织、岗位
edu-course      课程产品、班级实例、课次执行、扩展字段
edu-order       报名单、订单、履约确认、支付预留
edu-api         启动模块、Controller、接口聚合
```

后续建议增加：

```text
edu-student     学员、家长、联系人
edu-staff       教师、员工、岗位
edu-schedule    排课、调课、停课、补课
edu-attendance  考勤、请假、补签、统计
edu-finance     收款、退款、合同、发票、对账
edu-report      看板、报表、导出任务
edu-file        上传、导入、导出、对象存储
edu-notice      站内信、短信、微信模板消息
edu-audit       操作日志、登录日志、数据变更日志
```

## 高并发设计

- 登录态、权限、租户配置进入 Redis 缓存。
- 报表、Excel 导入导出、批量通知走异步任务。
- 考勤保留明细表，同时维护日、月维度汇总表。
- 关键写接口做幂等控制，防止重复提交。
- 重要查询必须包含 `tenant_id`、业务主键和时间范围索引。
- 后台任务避免直接阻塞 HTTP 请求。

## 日志追踪

每个 HTTP 请求都会生成或继承一个请求 ID：

```text
X-Request-Id
```

后端会把它写入：

- 响应头
- 统一响应体 `requestId`
- 日志 MDC 字段 `%X{requestId}`

线上排查问题时，可以让前端把接口返回的 `requestId` 贴给后端，后端直接按这个 ID 搜索日志。

## 数据隔离

第一阶段使用共享库共享表：

```text
tenant_id + 数据权限条件
```

这种方案成本低，适合 SaaS 初期快速扩展。后续可支持大客户独立库或独立部署。
