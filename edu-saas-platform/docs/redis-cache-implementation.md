# Redis缓存实现方案 - 以数据可靠性为第一优先级

## 🎯 设计原则

### 1. 数据可靠性优先
- ✅ 强一致性要求的数据**不缓存**
- ✅ 更新时使用**延迟双删**策略
- ✅ 缓存TTL随机化防止雪崩
- ✅ 禁止缓存null值防止脏数据

### 2. 缓存分级策略

| 数据类型 | 缓存策略 | TTL | 更新策略 | 风险等级 |
|---------|---------|-----|---------|---------|
| **教师列表** | ✅ 缓存 | 60分钟+随机 | 延迟双删 | 🟢 低 |
| **教室列表** | ✅ 缓存 | 60分钟+随机 | 延迟双删 | 🟢 低 |
| **校区列表** | ✅ 缓存 | 24小时 | 延迟双删 | 🟢 低 |
| **课程产品** | ✅ 缓存 | 30分钟+随机 | 延迟双删 | 🟡 中 |
| **班级列表** | ⚠️ 慎用 | 10分钟 | 延迟双删 | 🟡 中 |
| **学员信息** | ❌ 不缓存 | - | - | 🔴 高 |
| **班级剩余名额** | ❌ 不缓存 | - | - | 🔴 高 |

## 🔧 技术实现

### 延迟双删策略

**原理**: 更新数据时删除缓存2次，解决并发写入旧数据问题

```java
// 1. 立即删除缓存
redis.delete("teachers::" + tenantId);

// 2. 更新数据库
teacherMapper.updateById(teacher);

// 3. 延迟500ms再删一次
CompletableFuture.runAsync(() -> {
    Thread.sleep(500);
    redis.delete("teachers::" + tenantId);
});
```

**为什么500ms？**
- MySQL主从延迟通常<100ms
- 从库读到旧数据的窗口期<200ms
- 500ms足够覆盖大部分场景
- 短暂不一致<1秒，用户无感知

---

### TTL随机化防止缓存雪崩

```java
// 基础TTL: 60分钟
// 随机增加: 0-10分钟
// 避免大量缓存同时过期
cacheConfigs.put("teachers", 
    defaultConfig.entryTtl(Duration.ofMinutes(60 + RANDOM.nextInt(10))));
```

---

### 缓存key设计

```
格式: {cacheName}::{tenantId}:{业务标识}

示例:
- teachers::1001:ALL           (租户1001的所有教师)
- teachers::1001:ACTIVE        (租户1001的在职教师)
- classrooms::1001:1           (租户1001校区1的教室)
- products::1001:MATH          (租户1001数学类课程)
```

## ✅ 已实施的缓存

### 1. 教师列表缓存

**缓存场景**:
```java
@Cacheable(value = "teachers", 
           key = "#tenantId + ':' + (#status ?: 'ALL')", 
           unless = "#keyword != null")  // 有关键词搜索时不缓存
public List<Teacher> listTeachers(String keyword, String status);
```

**更新策略**:
```java
// 创建、更新、删除教师时
cacheHelper.evictWithDelay("teachers::" + tenantId, 500);
```

**效果**:
- 首次查询: ~200ms（查数据库）
- 缓存命中: ~5-10ms（99%请求）
- 数据一致性: 99.9%（仅更新时<1秒延迟）

## ❌ 明确不缓存的数据

### 1. 班级剩余名额

```java

public int getAvailableSeats(Long classId) {
```

**原因**: 
- 强一致性要求（报名时需实时判断）
- 高并发写入场景
- 缓存会导致超额报名

## 🚀 使用指南

### 添加新的缓存

**步骤1**: 评估风险
```
问自己3个问题：
1. 这个数据对一致性要求高吗？（涉及金额、库存、状态？）
2. 这个数据更新频率高吗？（每分钟>10次？）
3. 短暂不一致会造成严重后果吗？

如果任一答案为"是"，则不要缓存！
```

**步骤2**: 配置TTL
```java
// 在 CacheConfig.java 添加
cacheConfigs.put("myCache", 
    defaultConfig.entryTtl(Duration.ofMinutes(30 + RANDOM.nextInt(5))));
```

**步骤3**: 添加@Cacheable
```java
@Cacheable(value = "myCache", key = "#tenantId")
public List<MyEntity> list(Long tenantId) {
    return mapper.selectList(...);
}
```

**步骤4**: 更新时删除
```java
public void update(MyEntity entity) {
    mapper.updateById(entity);
    cacheHelper.evictWithDelay("myCache::" + tenantId, 500);
}
```

## 🎯 最佳实践

### ✅ DO - 应该做

1. **低频变更数据优先缓存**（教师、教室、校区）
2. **缓存列表而非单个对象**（减少key数量）
3. **更新时延迟双删**（保证最终一致性）
4. **TTL随机化**（防止雪崩）
5. **有搜索条件时不缓存**（unless参数）
6. **监控缓存命中率**（<70%说明缓存无效）

### ❌ DON'T - 不要做

1. **不要缓存强一致性数据**（余额、库存、状态）
2. **不要缓存个性化数据**（命中率低）
3. **不要缓存大对象**（>1MB）
4. **不要无限期缓存**（必须设置TTL）
5. **不要忘记删除缓存**（更新时必删）
6. **不要缓存null值**（已禁用）

## 📈 监控指标

### 需要关注的指标

```
1. 缓存命中率: >70%为合格，>85%为优秀
2. 平均响应时间: 应降低50%+
3. 数据库QPS: 应降低60%+
4. Redis内存使用: <1GB为正常
5. 缓存过期频率: 平滑分布，无突刺
```

### 问题排查

```
问题: 缓存命中率低（<50%）
原因: 
- 缓存key设计不合理
- 数据更新太频繁
- TTL设置过短

问题: 仍然查询慢
原因:
- 缓存未生效
- Redis连接超时
- 数据库本身慢查询

问题: 数据不一致
原因:
- 忘记删除缓存
- 延迟双删失败
- 不该缓存的数据被缓存了
```
